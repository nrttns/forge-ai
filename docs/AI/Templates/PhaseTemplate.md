Template Status: Canonical

---

# Purpose

This template defines the standard structure for every Phase document.

A Phase represents a major architectural capability.

A Phase owns architecture.

It does not own implementation.

---

# Required Structure

## 1. Status

Document status, phase identifier, framework, planning hierarchy, and authority.

## 2. Phase

Official Phase identifier, name, position, and architectural role.

## 3. Historical Capability Range

Historical capability range and immutable capability mapping.

## 4. Current Status

Current Stage, current Historical Capability, progress, and upcoming milestone.

## 5. Purpose

Why this Phase exists.

## 6. Vision

Long-term architectural vision.

## 7. Mission

How the Phase achieves its vision.

## 8. Architecture Position

Where the Phase sits in framework evolution.

## 9. Historical Context

Migration context, architectural motivation, and legacy considerations.

## 10. Framework Dependencies

Required previous Phases and future consuming Phases.

## 11. Phase Objectives

Measurable architectural objectives.

## 12. Architectural Responsibilities

What this Phase owns.

## 13. Architectural Boundaries

What this Phase does not own.

## 14. Architecture Stack

Layer or composition model introduced by this Phase.

## 15. Dependency Model

Allowed dependencies, forbidden dependencies, invariants.

## 16. Consumption Model

Who consumes this Phase and which contracts are exposed.

## 17. Ownership Model

Ownership matrix.

## 18. Stage Roadmap

Stage list with objective and capability mapping.

## 19. Historical Capability Mapping

Complete Stage-to-Capability mapping.

## 20. Allowed Work

Approved work categories.

## 21. Forbidden Work

Work that must not occur in this Phase.

## 22. Deliverables

Architecture, documentation, implementation, validation, and audits.

## 23. Entry Criteria

What must exist before the Phase begins.

## 24. Exit Criteria

Measurable completion requirements.

## 25. Phase Completion Certification

Checklist for certification.

## 26. Quality Gates

Mandatory validation, review, and architecture checks.

## 27. Related Audits

Relevant audits.

## 28. Related RFCs

Relevant RFCs.

## 29. Related Documents

References.

## 30. Next Phase

Next Phase and how it consumes this one.
