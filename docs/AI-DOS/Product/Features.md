# AI-DOS Features

AI-DOS features are contract-governed capabilities, not assumptions inferred from an agent host, programming language, or repository layout.

## Target-Aware Operation

AI-DOS resolves the invoked Target Project before operational work begins. The Target supplies its own identity, resources, protected areas, validation requirements, and execution authority.

## Product / Target Truth Separation

Reusable AI-DOS product contracts remain separate from Target-specific mission, planning, state, evidence, and implementation decisions. This allows the same product model to operate across different Target Projects without importing one project's truth into another.

## Deterministic Context Resolution

AI-DOS uses explicit invocation and Target declarations to resolve context. Missing, inaccessible, ambiguous, conflicting, unsupported, unresolvable, or out-of-boundary declarations become blockers rather than invitations to guess.

## Authority-Aware Routing

Instructions are routed through the applicable product, Target, governance, architecture, command, workflow, template, engine, and provider contracts. Higher authority is consumed rather than silently rewritten.

## Bounded Execution

Mutation requires explicit human intent, Target permission for the exact action, known ownership, preserved protected areas, declared validation, sufficient provider capability, and no unresolved blocker.

## Human Governance

Human Governance remains final. AI-DOS does not treat model confidence, implementation capability, validation success, review outcome, repository activity, or merge state as automatic approval.

## Provider Neutrality

AI-DOS does not depend on one model vendor, agent host, editor, repository provider, or tool protocol. Providers expose capabilities and execute bounded contracts; they do not own product or Target truth.

## Safe Stop

When required context, authority, ownership, integrity, compatibility, or permission cannot be resolved, AI-DOS stops before mutation and reports the blocker.

## Evidence-Backed Completion

Completed work reports the actual scope, changed resources, authorities consumed, validation performed, unresolved risks, lifecycle status, and one recommended bounded next action.

## Lifecycle Separation

AI-DOS keeps execution, validation, review, certification, Human Governance approval, promotion, and release distinct. Evidence from one stage does not silently grant another stage.

## MD-First System Contracts

Markdown artifacts are the canonical normative system surface. Existing code and package tooling are bounded implementation surfaces and cannot override the governing contracts.

## Distribution Awareness

When release, packaging, installation, update, rollback, uninstall, integrity, or discovery is in scope, AI-DOS routes through its distribution authority. Installation alone does not grant permission to mutate a Target Project.

## Current Capability Disclaimer

A documented architectural capability is not necessarily fully implemented, certified, released, or active. Protected future capabilities remain inactive until their own dependencies, validation, review, and Human Governance gates are satisfied.
